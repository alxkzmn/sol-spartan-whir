# sol-spartan-whir

Solidity verifier for the Spartan-WHIR SNARK over the KoalaBear field (p = 2^31 - 2^24 + 1).

This repository contains the Foundry verifier project. Test fixtures in `testdata/` are checked in and generated by the companion Rust exporter, [spartan-whir-export](https://github.com/alxkzmn/spartan-whir-export), from the Rust [Spartan-WHIR implementation](https://github.com/alxkzmn/spartan-whir).

## Build and test

```sh
forge build
forge test
```

Test fixtures in `testdata/` are checked in, so the test suite runs without any external setup. To regenerate them, point `SPARTAN_WHIR_EXPORT_DIR` at a checkout of [spartan-whir-export](https://github.com/alxkzmn/spartan-whir-export) and invoke the exporter:

```sh
cargo run --release --manifest-path "$SPARTAN_WHIR_EXPORT_DIR/Cargo.toml" --bin export-fixtures -- testdata
```

The exporter depends on the Rust [Spartan-WHIR implementation](https://github.com/alxkzmn/spartan-whir), which is the source of truth for the protocol; the Solidity verifier in this repository must stay byte-compatible with it.

See [AGENTS.md](./AGENTS.md) for the full command catalog: schedule-specific fixture exporters, gas profiling and flamegraph workflows, transaction-gas benchmarking, the precompile-runner Foundry fork ([alxkzmn/foundry](https://github.com/alxkzmn/foundry/tree/codex/ext8-precompile-runner)), and the upstream verifier source anchors.

### Quintic software profile

The current quintic build target was profiled with the native blob verifier flamegraph harness.

The canonical Foundry gas for this path is `5,526,933`. The verifier call itself, excluding fixture-loading harness noise, accounts for `4,768,744` gas.

| Profile bucket                                                             |         Gas | Share of verifier call |
| -------------------------------------------------------------------------- | ----------: | ---------------------: |
| Inlined verifier body, mostly fused STIR row evaluation and blob traversal | `3,290,290` |                `67.8%` |
| Final STIR and final-polynomial checks                                     |   `655,138` |                `13.5%` |
| Explicit ext5 `mul` / `square` / `sub` functions                           |   `283,884` |                 `5.8%` |
| Final value evaluation                                                     |   `134,938` |                 `2.8%` |
| Base-field powers and query-point generation                               |   `141,586` |                 `2.9%` |
| Equality terms and dim-4 equality-weight precomputation                    |   `160,962` |                 `3.3%` |
| Merkle frontier computation                                                |   `104,935` |                 `2.2%` |
| STIR query sampling and transcript sampling                                |   `112,117` |                 `2.3%` |
| Extension inversion or division                                            |         `0` |                 `0.0%` |

The profile does not show the fused row dot-product kernels as separate function calls because `via_ir` inlines them into the verifier body. The dominant arithmetic work is therefore repeated WHIR row and final evaluation, with extension multiplication embedded inside fused multiply-accumulate kernels. It is not extension inversion or division, and it is no longer mostly standalone calls to `KoalaBearExt5.mul`.

#### Arithmetic and proof-structure differences

The execution-gas gap comes from both arithmetic cost and proof-structure differences. The table below lists the benchmark parameters that matter most.

| Item                               | sol-spartan-whir | sol-whir |
| ---------------------------------- | ---------------- | -------- |
| **Field / arithmetic**             | KoalaBear + ext4 | BN254    |
| **Configured WHIR rounds**         | 2                | 3        |
| **Per-round query counts**         | 9, 6             | 9, 6, 5  |
| **Final query count**              | 5                | 4        |
| **Total STIR queries incl. final** | 20               | 24       |
| **Commitment OOD samples**         | 2                | 1        |
| **Final sumcheck rounds**          | 4                | 0        |

Implications:

- `sol-whir` has one extra configured round and `4` more total STIR queries.
- `sol-spartan-whir` pays for more OOD sampling and a real final sumcheck.
- The one structural delta we can price directly is the final sumcheck: ~`27.1k` gas in `sol-spartan-whir`, while the `sol-whir` benchmark config has `finalSumcheckRound = 0`.
- Proof structure explains part of the gap; the rest is execution cost inside the verifier.

### Extension degree choice for k=22 / JohnsonBound 100-bit

The quintic extension is the current high-security build target. Quartic (~124 field bits) does not have the soundness budget for `num_variables = 22` at JohnsonBound 100-bit without large grinding caps or query counts that land in a high-gas regime; octic (~248 field bits) has more headroom than needed and costs more per query. Quintic (~155 field bits) sits in the middle: it supports `oodSamples = 1` at this security target with extension arithmetic cost closer to quartic than to octic. In this README, quartic is retained as the 80-bit baseline for the `sol-whir` comparison, not as the high-security deployment target.

#### Applied octic optimizations

- The octic contract is `WhirBlobVerifierNative8_k22_jb100_lir6_ff4_rsv1`. Its external entrypoint is `verify(bytes32 expectedCommitment, bytes blob)`, and it reads the proof blob directly from calldata.
- The STIR row kernel is specialized for `rowLen=16` and uses the combined blob-side hash-and-evaluate path with packed 20-byte frontier reduction.
- Ext8 folding uses direct arithmetic kernels, including the Karatsuba-based `_foldOnceWithCoeffs(...)` implementation used in the contract.
- Round-constraint accumulation uses fixed `18/14/10` select kernels and a fused ext8 equality-term path instead of the generic ext8 multiplication helpers in the shared verifier code.
- Final STIR uses two fixed five-query checks for the `64`-coefficient final polynomial.
- Final-value evaluation uses a fixed scratch-memory fold tree instead of a dynamic `evals` array.
- Extension-row STIR dot products use lazy reduction across the 16 row values, avoiding a chain of packed ext8 multiply/add operations.
- [whir_param_sweep.py](./whir_param_sweep.py) models this octic configuration, but its octic gas calibration row still uses `7,383,992`. The measured native blob gas is `6,908,778`, so the octic model row is not an exact anchor.

### Gas accounting

**Execution gas** is measured via the Foundry gas tests for the deployed contract entrypoints.

**Total tx gas** is measured from real transaction receipts, then split as:

```text
total tx gas = intrinsic gas (21,000) + calldata gas + execution remainder
```

where:

- `calldata gas = 4 * zero_bytes + 16 * nonzero_bytes`
- `execution remainder = receipt.gasUsed - 21,000 - calldata gas`

The quartic comparison calldata footprint comes from `script/WhirBlobNativeTxBenchmark_lir6_ff5_rsv1.s.sol`. The documented octic production calldata footprint comes from `script/WhirBlobNativeTxBenchmark_k22_jb100_lir6_ff4_rsv1.s.sol`. See the [`tx-gas-benchmarking` skill](./.agents/skills/tx-gas-benchmarking/SKILL.md) for the measurement workflow.

For `sol-whir`, the tx numbers come from the [broadcast artifact committed in privacy-ethereum/sol-whir](https://github.com/privacy-ethereum/sol-whir/blob/main/broadcast/Verify.s.sol/31337/run-latest.json). The `sol-whir` benchmark harness does not compile under the toolchain used for this measurement (`stack too deep` / Yul stack-too-deep), so that committed measurement remains the source of truth.

`sol-spartan-whir` Foundry build settings for the quartic comparison contract are `via_ir = true`, `optimizer = true`, and `optimizer_runs = 833`. The recorded `WhirVerifier4` deployed bytecode size is `24,008` bytes.
